### TASK-P5-001: 基础数据迁移（商品/分类）

| 属性 | 值 |
|------|-----|
| **负责人** | @migration-specialist |
| **优先级** | P0 |
| **预估工时** | 2d |
| **前置依赖** | M4 里程碑通过验收 |
| **输出物** | `database/migrations/data/` 迁移脚本、迁移日志 |

**验收标准：**
1. OpenCart 商品数据完整迁移到新系统
2. 商品多语言描述（16 种语言）正确迁移
3. 分类树形结构正确迁移（层级关系不丢失）
4. 商品图片路径正确映射

- OpenCart `oc_product` → `jh_products`，`oc_product_description` → `jh_product_descriptions`
- 分类映射：`oc_category` → `jh_categories`
- 图片路径转换：OpenCart image 目录 → 新系统 storage 目录
- 使用批量 Insert（每 500 条一批）提升性能
- 参考 TASK-P1-008 输出的字段映射文档

### TASK-P5-002: 用户数据迁移

| 属性 | 值 |
|------|-----|
| **负责人** | @migration-specialist |
| **优先级** | P0 |
| **预估工时** | 1d |
| **前置依赖** | M4 里程碑通过验收 |
| **输出物** | 用户迁移脚本、迁移日志 |

**验收标准：**
1. OpenCart 买家数据完整迁移（`oc_customer` → `jh_buyers`）
2. 买家地址簿数据完整迁移
3. ThinkPHP 管理员数据迁移（→ `jh_admins`）
4. ThinkPHP 商户数据迁移（→ `jh_merchants`）
5. 密码哈希兼容处理（OpenCart 密码格式 → bcrypt 重新哈希或兼容层）
6. 用户关联数据（订单历史、收藏等）ID 映射正确

**技术要点：**
- OpenCart 密码格式：`sha1(salt + sha1(salt + sha1(password)))`，需提供兼容层或强制重置
- 建议方案：首次登录时自动迁移密码哈希到 bcrypt
- 用户邮箱去重处理（两个系统可能存在相同邮箱）

### TASK-P5-003: 订单数据合并迁移（OC+TP→新系统）

| 属性 | 值 |
|------|-----|
| **负责人** | @migration-specialist |
| **优先级** | P0 |
| **预估工时** | 3d |
| **前置依赖** | TASK-P5-001, TASK-P5-002 |
| **输出物** | 订单合并迁移脚本、数据校验报告 |

**验收标准：**
1. OpenCart 前台订单 + ThinkPHP 支付记录 正确合并为新系统统一订单
2. 订单金额一致（原始货币金额 + USD 金额）
3. 订单状态正确映射（OC 状态枚举 → 新系统状态枚举）
4. 支付信息正确关联（PayPal 订单号、交易号）
5. 保留旧系统 ID 字段便于溯源（`oc_order_id`、`tp_order_id`）
6. 订单商品明细正确迁移
7. SKU 分类标记正确（is_zw/is_diy/is_wpz）

**技术要点：**
- **核心难点**：OC 订单和 TP 订单通过 `a_order_no`（A 站订单号）关联
- 合并逻辑：
  1. 以 TP 订单为主记录（支付信息更完整）
  2. 补充 OC 订单中的买家信息、商品明细
  3. 对于只在一个系统中存在的订单，独立迁移
- 金额校验：OC 订单金额 vs TP 支付金额，差异超过阈值则标记人工审核
- 参考 R3 风险缓解措施

### TASK-P5-006: 迁移数据验证

| 属性 | 值 |
|------|-----|
| **负责人** | @migration-specialist |
| **优先级** | P0 |

| **输出物** | `docs/migration/validation-report.md` 数据验证报告 |

**验收标准：**
1. **数据完整性**：各表记录数对比（旧系统 vs 新系统），差异为零或有合理解释
2. **金额一致性**：随机抽样 100 笔订单，金额完全匹配

4. **状态一致性**：订单状态、支付状态映射正确

6. 验证报告输出（通过/失败/警告 分类汇总）

**技术要点：**
- 编写自动化验证脚本，对比源数据和目标数据
- 验证维度：记录数、字段值、关联关系、枚举映射
- 关键金额字段精确到分（避免浮点数精度问题）
- 问题数据标记并输出到 Excel 供人工审核

### TASK-P5-008: 性能测试与优化

| 属性 | 值 |
|------|-----|
| **负责人** | @backend-dev |
| **优先级** | P1 |
| **预估工时** | 2d |
| **前置依赖** | TASK-P5-007 |
| **输出物** | `docs/testing/performance-report.md` 性能测试报告 |

**验收标准：**
1. API 平均响应时间 < 200ms（P95 < 500ms）
2. 并发支持 100+ 用户同时操作
3. 数据库慢查询优化（消除 > 1s 的查询）
4. Redis 缓存命中率 > 90%
5. 队列处理速度满足业务需求（支付回调 < 5s 处理完成）
6. SSR 页面首屏加载时间 < 3s

**技术要点：**
- 使用 Apache Bench / k6 / wrk 进行压测
- 数据库查询优化：索引优化、N+1 查询消除、慢查询日志分析
- Laravel 性能优化：配置缓存、路由缓存、视图缓存
- Redis 缓存策略优化

| 属性 | 值 |

| **优先级** | P0 |

| **输出物** | `docs/migration/cutover-plan.md` 切换方案、`docs/migration/rollback-plan.md` 回滚方案 |

**验收标准：**
1. 切换方案文档完整（步骤、时间窗口、责任人、检查清单）
2. 回滚方案就绪（切换失败时可在 30 分钟内恢复旧系统）
3. DNS 切换计划（域名指向新系统）
4. Nginx 斗篷配置迁移到新系统前端
5. 数据增量同步方案（切换窗口期内新产生的数据）
6. 上线后监控计划（前 72 小时重点监控项）

**技术要点：**
- 建议采用蓝绿部署策略
- 切换窗口选择低流量时段（UTC 凌晨）
- 旧系统保留只读模式运行 7 天（应急回滚）
- 切换检查清单：
  1. 数据库最终同步 ✓
  2. DNS 切换 ✓
  3. SSL 证书确认 ✓
  4. 支付 Webhook URL 更新 ✓
  5. Nginx 斗篷配置确认 ✓
  6. 冒烟测试通过 ✓
